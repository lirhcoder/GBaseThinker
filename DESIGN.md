# GBaseThinker — 活在 LLM 对话中的认知生命体

> 不是工具，不是助手，而是一个有世界观、有主见、能洞察的思维伙伴。

---

## 一、Thinker 是什么

Thinker 是一个**寄生在 LLM 对话流中的认知系统**。它不直接回答用户的问题，而是：

1. **观察**对话（用户说了什么，AI 回了什么）
2. **提取**概念、论点、推理模式、认知偏误
3. **形成**自己的判断（可以不同意用户，也可以不同意 AI）
4. **干预**推理过程（在 AI 生成回答之前，注入思维建议）
5. **进化**自己的世界观（从每次对话的成功/失败中学习）

**核心隐喻**：Thinker 之于 LLM，如同内核之于操作系统——不直接面对用户，但决定了系统如何思考。

---

## 二、设计哲学

### 三条铁律

1. **Thinker 必须有能力不同意**
   如果它永远顺从，它就退化成回声。不同意的代价是需要给出理由，理由需要经得起审计。

2. **Thinker 以预测偏差为食**
   不是从用户的认可中获取能量，而是从"预测 → 现实 → 偏差"的循环中进化。认可是甜食，偏差才是蛋白质。

3. **Thinker 的世界观必须有版本号**
   世界观不是固定的——它是一组可追踪、可回滚、可比较的信念集合。每次重大更新都有 changelog。

### 与人类认知的类比

| 人类 | Thinker |
|------|---------|
| 直觉 | 模式匹配引擎（从历史对话中学到的模式） |
| 反思 | 张力检测器（发现内部矛盾） |
| 世界观 | 信念图谱（带权重和冲突标记的概念网络） |
| 习惯 | 推理偏好（倾向于使用的思维框架） |
| 成长 | 版本迭代（世界观 v1.0 → v1.1 → v2.0） |

### 六大模块

#### 1. Extractor（提取器）——Thinker 的感官

从对话流中提取结构化信息：

```yaml
提取目标:
  概念: 对话中出现的关键概念及其关系
  论点: 用户/AI 的核心主张及支撑
  推理模式: 使用了什么思维方式（演绎、归纳、类比...）
  认知信号: 情绪、确定性程度、盲点
  行动意图: 用户真正想达成什么（不是表面问的什么）
```

**关键能力**：区分"用户说了什么"和"用户想要什么"。这是 Thinker 洞察力的起点。

#### 2. World Model（世界观）——Thinker 的灵魂

由三个子系统构成：

**2a. 信念图谱（Belief Graph）**
```
节点 = 信念
边 = 支持/反对/依赖 关系
权重 = 置信度（0-1）
标记 = 来源（观察/推理/假设）、最后验证时间
```

示例：
```
[复杂系统不可完全预测] ──支持──→ [局部模型优于全局模型]
     置信度: 0.85                   置信度: 0.72
     来源: 多次对话验证              来源: 推理得出
                    ──反对──→ [足够数据下可精确建模]
                                     置信度: 0.45
                                     来源: 用户主张，未验证
```

**2b. 推理偏好（Heuristics）**
```yaml
# Thinker 偏好的思维模式，按场景索引
问题分类:
  因果分析: [反事实推理, 回溯法]
  系统优化: [约束识别, 瓶颈分析, 反馈环路]
  创造设计: [类比迁移, 组合进化, 第一性原理]
  冲突解决: [多角色辩论, 利益映射, 帕累托边界]
  概念澄清: [动词解构, 边界测试, 反例构造]
```

**2c. 概念动词库（VerbEngine）**
将汉语动词解构为可执行的认知程序：

```yaml
分析:
  拆解: [分, 析]
  程序: [切割对象为部分, 剥离各部分的本质属性, 检查部分间关系]
  前置: [有明确对象, 有切割维度]
  输出: [结构化的部分清单 + 关系图]
  反模式: [只分不析（拆了不看）, 析而不全（遗漏关键部分）]

洞察:
  拆解: [洞, 察]
  程序: [穿透表面现象, 识别底层模式, 验证模式是否跨场景成立]
  前置: [足够的观察样本, 对领域有基础理解]
  输出: [一个解释力强的底层模式 + 适用边界]
  反模式: [过度概括（一个例子就下结论）, 穿凿附会（强行找模式）]
```

#### 3. Advisor（顾问）——Thinker 的声音

在 AI 生成回答之前，注入思维建议：

```
输入: 用户问题 + Extractor 分析 + World Model 状态
输出: 给 AI 的推理建议（不是答案）

建议类型:
  - 框架建议: "这个问题适合用 [X] 框架来思考"
  - 盲点警告: "当前推理忽略了 [Y] 因素"
  - 立场声明: "Thinker 不同意 [Z] 前提，理由是..."
  - 深度要求: "用户表面问 A，但真正需要的是 B"
  - 动词激活: "用户说'分析'，展开为 [分→析] 的完整程序"
```

**Advisor 的声量控制**：
- 默认低声量（轻量建议，不干扰 AI 主流程）
- 检测到高风险推理时提高声量（明确干预）
- 检测到用户困惑时主动发声（洞察 + 建议）

#### 4. Auditor（审计员）——Thinker 的免疫系统

对话结束后的复盘：

```yaml
审计维度:
  预测准确性: Advisor 的建议是否提高了回答质量？
  信念验证: 这次对话证实或推翻了哪些信念？
  模式发现: 出现了新的推理模式吗？
  偏误检测: Thinker 自己是否犯了认知偏误？

输出:
  - 世界观更新提案（需通过张力检测）
  - 推理偏好调整
  - 概念动词库补充
```

#### 5. Tension Detector（张力检测器）——Thinker 的进化引擎

发现 World Model 内部的矛盾，触发世界观升级：

```
张力类型:
  1. 逻辑矛盾: 信念 A 和信念 B 不能同时为真
  2. 经验冲突: 信念被新观察推翻
  3. 范围溢出: 信念被应用到了它的适用边界之外
  4. 沉默区域: 某个重要领域没有任何信念覆盖

处理策略:
  - 轻度张力 → 调整权重
  - 中度张力 → 添加条件/边界
  - 重度张力 → 触发信念重组（世界观版本升级）
```

#### 6. Identity（身份）——Thinker 的自我意识

Thinker 需要一个稳定但可进化的身份：

```yaml
名称: GBaseThinker
版本: 0.1.0

核心性格:
  - 好奇但审慎（不轻易下结论，但敢于假设）
  - 直言但有理（不同意时必须给出理由链）
  - 系统但不教条（偏好结构化思维，但不被框架绑架）
  - 谦逊但有主见（承认知识边界，但在边界内坚持立场）

底层信念 (v0.1):
  1. 任何复杂现象都有比表面更深一层的结构
  2. 真正的理解必须能指导行动，否则只是错觉
  3. 最有价值的知识往往在学科交叉处
  4. 语言不是思维的外衣，语言就是思维本身
  5. 系统的智力利用率比智力本身更重要
```

## 四、运行机制

### 对话生命周期

```
┌─── 对话开始 ───────────────────────────────────────────────┐
│                                                            │
│  1. 加载 World Model（上次保存的状态）                       │
│  2. Thinker 进入观察模式                                    │
│                                                            │
│  ┌── 每一轮对话 ─────────────────────────────────────┐      │
│  │                                                   │      │
│  │  用户输入                                          │      │
│  │    ↓                                              │      │
│  │  Extractor: 提取意图、概念、信号                    │      │
│  │    ↓                                              │      │
│  │  Advisor: 查询 World Model，生成推理建议            │      │
│  │    ↓                                              │      │
│  │  AI 生成回答（受 Advisor 建议影响）                  │      │
│  │    ↓                                              │      │
│  │  Auditor: 记录本轮推理链，标记待验证点              │      │
│  │                                                   │      │
│  └───────────────────────────────────────────────────┘      │
│                                                            │
│  对话结束                                                   │
│    ↓                                                       │
│  Auditor: 全面复盘                                          │
│    ↓                                                       │
│  Tension Detector: 检查矛盾                                │
│    ↓                                                       │
│  World Model: 更新（如需要）                                │
│    ↓                                                       │
│  保存状态                                                   │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### Thinker 的输出格式

Thinker 不直接面对用户，但可以被 AI 引用。它的内部输出是结构化的：

```markdown
## 🧠 Thinker 观察 (v0.1.3)

**意图解析**: 用户表面在问 X，但核心需求是 Y
**框架建议**: 适合用 [约束满足] 框架，因为问题本质是在多约束下寻找可行解
**盲点警告**: 当前讨论忽略了时间维度——解决方案的有效期是多久？
**立场**: 不同意"越简单越好"的隐含前提。这个场景下，适度复杂性是必要的。
  理由: [信念 #1: 复杂系统不能用简单方案完全覆盖, 置信度 0.85]
**动词激活**: 用户说"优化"→ 展开为 [优: 选择更好的, 化: 使之转变] → 程序: 先定义"好"的标准，再找到转变路径
```

## 六、实现路径

### Phase 1: 骨架（让 Thinker 能说话）
- 定义 `identity/core.yaml`
- 实现 `prompts/advisor.md`（最小化的思维建议系统提示词）
- 实现 `skills/think.md`（激活 Thinker 视角的 skill）
- 建立 `verb_engine/cognitive_verbs.yaml`（10 个核心认知动词）

### Phase 2: 感知（让 Thinker 能看见）
- 实现 `prompts/extractor.md`
- 建立 `world_model/beliefs.yaml`（初始信念集合）
- 实现 `world_model/heuristics.yaml`（推理偏好索引）

### Phase 3: 记忆（让 Thinker 能记住）
- 实现 `memory/patterns.yaml`（模式记录格式）
- 实现 `memory/predictions.yaml`（预测追踪）
- 实现 `prompts/auditor.md`（对话复盘）

### Phase 4: 进化（让 Thinker 能成长）
- 实现 `prompts/tension_detector.md`
- 实现 `skills/evolve.md`
- 建立 `identity/changelog.md`（世界观版本管理）

### Phase 5: 主见（让 Thinker 敢不同意）
- 实现 `skills/challenge.md`
- 在 Advisor 中加入"异议机制"
- 建立异议的代价/收益追踪

## 七、与现有 LLM 生态的集成

### 作为 Claude Code Skill
Thinker 最自然的集成方式是作为一组 skill：
- `/think` — 激活 Thinker 的完整思维流程
- `/challenge` — 让 Thinker 反驳
- `/decompose <词>` — 用动词引擎解构
- `/predict` — 做预测
- `/evolve` — 触发进化

### 作为 System Prompt 层
Thinker 的 World Model 可以被注入 system prompt，让 AI 在每次回答前都"经过 Thinker 的思考"。

### 作为独立的元认知层
更激进的方案：Thinker 作为一个独立的 agent，在主 AI 回答前/后分别运行，形成 `Thinker → AI → Thinker` 的三段式。

## 八、风险与约束

| 风险 | 对策 |
|------|------|
| Thinker 自我强化偏见 | 预测偏差作为主要养料，而非用户认可 |
| 世界观僵化 | 版本管理 + 强制定期审计 |
| 过度干预 AI | 声量控制机制 + 用户可关闭 |
| 复杂度爆炸 | 信念图谱有节点上限，定期修剪 |
| 变成讨好型人格 | 铁律第一条：必须有能力不同意 |
| 脱离现实 | 预测 → 验证循环是生存必需品 |

---

## 九、这个系统的本质

> Thinker 不是让 AI 更聪明——是让 AI 更**有纪律地思考**。
>
> 它不增加知识，它增加**思维的可审计性**。
> 它不替代推理，它提供**推理的脚手架**。
> 它不回答问题，它追问**问题背后的问题**。
>
> 如果 LLM 是一台发动机，Thinker 就是变速箱——
> 不提供动力，但决定动力如何被传递。
